# 01 · 整体架构

> 先把"这是个什么东西、怎么跑起来的"讲清楚，后面每个功能才好懂。

## 1. 一句话定位

星港是一个 **全栈单体（full-stack monolith）Web 应用**：用户看的页面、处理逻辑的后端、读写的数据库，全在**同一个项目、同一个 Node 进程**里。没有"前端项目 + 后端项目"的拆分。

为什么这么选？因为它让一个人就能快速把整个产品做出来，而且 Next.js 这个框架天生就是干这个的。

## 2. 技术栈（每个是干嘛的）

| 层 | 用的技术 | 通俗解释 |
|---|---|---|
| 框架 | **Next.js 16（App Router）** | 一个"既能做前端页面、又能做后端接口"的 React 框架。它是整个项目的骨架。 |
| UI 库 | **React 19** | 用组件搭页面。组件 = 可复用的 UI 积木。 |
| 样式 | **Tailwind CSS 4** | 用 `class="px-4 text-sm"` 这种工具类直接写样式，不用单独写 CSS 文件。 |
| 动效 | **framer-motion** | 做淡入、滑动这种动画。 |
| 数据库 | **PostgreSQL** | 存所有数据的关系型数据库（表、行、列）。 |
| 数据库工具 | **Prisma 7** | ORM。用 `prisma.user.findMany()` 这种代码查库，不用手写 SQL。 |
| 登录 | **iron-session** | 把登录态加密塞进浏览器 Cookie。 |
| 语言 | **TypeScript** | 带类型检查的 JavaScript，写错类型编译就报错。 |
| 测试/验证 | **Playwright** | 用代码驱动真实浏览器（Chrome）点页面、截图，做端到端验证。 |

## 3. Next.js App Router 的核心心智模型（最重要）

这是理解整个项目的钥匙。Next.js 的 App Router 把组件分成两种：

### 服务端组件（Server Component）—— 默认就是它

```
src/app/page.tsx        ← 商店首页，是服务端组件
src/app/library/page.tsx ← 库页面，是服务端组件
```

特点：
- **在服务器上运行**，不在浏览器跑。
- 可以**直接 `await` 查数据库**（因为它本来就在服务器上，离数据库很近）。
- 把渲染好的结果发给浏览器，浏览器**不用为它下载任何 JS**——所以首屏快。
- 但它**不能交互**：没有 `useState`、没有 `onClick`、不能拖动。

例子（真实代码，`src/app/page.tsx`）：

```tsx
export default async function StorePage() {
  // 直接在组件里 await 查数据——这在传统前端是做不到的
  const featured = await getFeatured();   // 内部是 prisma 查询
  return <HeroCarousel products={featured} />;
}
```

### 客户端组件（Client Component）—— 需要交互时才用

文件顶部写 `"use client"`，例子：聊天窗、好友面板、轮播图、可拖动窗口。

```
src/components/social/social-layer.tsx  ← "use client"，因为要处理点击、轮询、拖动
src/components/store/hero-carousel.tsx  ← "use client"，因为要自动切换
```

特点：
- **会被打包成 JS 发到浏览器**，在浏览器里运行。
- 能用 `useState`（记住状态）、`useEffect`（做定时器/订阅）、`onClick`、拖动等。
- 不能直接查数据库（它在浏览器里，离数据库很远）——要数据得通过 **Server Action**。

### 两者怎么配合

典型套路：**服务端组件查好初始数据 → 当成 props 传给客户端组件 → 客户端组件负责后续交互**。

真实例子（`src/app/layout.tsx`，每个页面都会经过的根布局）：

```tsx
export default async function RootLayout({ children }) {
  const user = await getCurrentUser();              // 服务端：查当前用户
  const friends = await getFriendsWithPresence();   // 服务端：查好友列表
  return (
    <body>
      {children}
      {/* 把服务端查到的好友当初始数据，交给客户端组件去做实时交互 */}
      <SocialLayer me={...} initialFriends={friends} initialRequests={requests} />
    </body>
  );
}
```

> **面试金句**：服务端组件负责"首屏直出数据"，客户端组件负责"之后的交互"，两者用 props 衔接——这是 App Router 的核心范式。

## 4. 写数据：Server Actions（替代手写 REST 接口）

传统做法：前端 `fetch('/api/sendMessage')` → 后端写一个 `/api/sendMessage` 路由处理。要写两边，还要约定 URL、请求体格式。

Next.js 的 Server Action 把这件事**简化成"调用一个函数"**：

```ts
// src/app/friends-actions.ts —— 文件顶部 "use server" 标明这些是服务端函数
"use server";
export async function sendMessageAction(handle: string, body: string) {
  return sendMessage(handle, body);   // 内部写库
}
```

```tsx
// 客户端组件里，像调本地函数一样调它：
import { sendMessageAction } from "@/app/friends-actions";
await sendMessageAction("linyuan", "你好");   // 框架自动发了一个 POST 请求到服务器
```

你**没写任何 URL、没写任何 fetch**，框架在背后帮你做了网络请求 + 序列化 + 反序列化。这就是 Server Action。

> **面试金句**：Server Action 让"前端调后端"变成"调一个 async 函数"，省掉了手写 REST 接口和 fetch 的样板代码；它本质还是一次 HTTP POST，只是框架替你封装了。

## 5. 代码分层（项目目录怎么组织的）

```
src/
├── app/                    # 页面 + Server Actions（路由由文件夹结构决定）
│   ├── page.tsx            #   /            商店首页
│   ├── library/page.tsx    #   /library     库
│   ├── p/[slug]/page.tsx   #   /p/xxx       产品详情（[slug] 是动态参数）
│   ├── u/[handle]/page.tsx #   /u/xxx       个人主页
│   ├── settings/...        #   /settings/*  设置
│   ├── api/v1/...          #   /api/v1/*    给外部应用的 REST 接口
│   └── *-actions.ts        #   Server Actions（写操作入口）
│
├── components/             # UI 组件（服务端 + 客户端）
│   ├── social/             #   好友/聊天（客户端，交互重）
│   ├── store/              #   商店首页各区块
│   └── ui/                 #   通用小组件（头像、徽章、评分）
│
└── lib/                    # 业务逻辑层（"service"，纯服务端）
    ├── db.ts               #   Prisma 客户端单例
    ├── session.ts          #   登录态读写
    ├── catalog.ts          #   产品目录查询（页面只从这取数）
    ├── message-service.ts  #   聊天消息逻辑
    ├── gateway-core.ts     #   AI Gateway 代理核心
    └── ...                 #   每个领域一个 service 文件
```

**关键设计原则：页面/组件不直接写 Prisma 查询，而是调用 `lib/` 下的 service 函数。** 好处是：
- 业务逻辑集中在一处，页面只管展示。
- 比如所有页面取产品都走 `catalog.ts`，将来换数据源只改这一个文件。

> **面试金句**：我做了"门面（facade）"分层——`lib/catalog.ts` 是产品数据的唯一入口，页面不碰数据库细节。早期是假数据，后来内部换成 Prisma，页面一行没改。

## 6. 数据库放在哪

本机一个**独立的 PostgreSQL 实例**，数据存在项目里的 `.pgdata/` 文件夹，跑在 **5544 端口**（避开系统默认的 5432）。用 `pnpm db:start` / `db:stop` 启停。连接串放在 `.env` 里（不进 git）。

这样做的好处：项目自带数据库，不依赖你电脑上别的数据库配置，换台机器也能跑。

## 7. 一次请求的完整生命周期（串起来）

以"打开库页面"为例：

```
浏览器请求 /library
   │
   ▼
Next.js 服务器收到 → 运行 library/page.tsx（服务端组件）
   │
   ├─ await getSessionUserId()  → 读 Cookie 解出"我是谁"
   ├─ await getLibrary()        → Prisma 查 PostgreSQL 拿到我的库
   │
   ▼
在服务器上把数据渲染成 HTML
   │
   ▼
HTML 发回浏览器（首屏直接有内容，SEO 友好、快）
   │
   ▼
浏览器再加载客户端组件的 JS（如好友坞 SocialLayer）→ 页面变得可交互
```
