# 05 · 数据库与数据建模

> 讲清楚：数据怎么存、表怎么设计、为什么有些字段"故意冗余"、迁移和种子踩的坑。
> 涉及文件：`prisma/schema.prisma`、`prisma/seed.ts`、`src/lib/db.ts`。

---

## 1. 用什么、怎么用

- **PostgreSQL**：关系型数据库。数据存成"表（table）+ 行（row）+ 列（column）"，表之间有外键关联。
- **Prisma（ORM）**：ORM = Object-Relational Mapping，"对象关系映射"。让你**用写 TypeScript 的方式操作数据库**，不用手写 SQL：

```ts
// 不用写 SQL：SELECT * FROM "User" WHERE handle = 'me'
const me = await prisma.user.findUnique({ where: { handle: "me" } });
```

好处：有类型检查（查错字段编译就报错）、自动补全、跨数据库可移植。

### Prisma 客户端单例（`src/lib/db.ts`）
数据库连接很贵，不能每次请求都新建。所以全局只建一个 Prisma 实例，挂在 `globalThis` 上复用：

```ts
const globalForPrisma = globalThis as { prisma?: PrismaClient };
export const prisma = globalForPrisma.prisma ?? buildClient();
if (process.env.NODE_ENV !== "production") globalForPrisma.prisma = prisma;
```

> **为什么挂 globalThis？** 开发模式下代码会热重载，每次重载都新建连接会很快耗尽数据库连接数。挂在全局变量上能跨重载复用同一个实例。

## 2. Schema：表是怎么设计的（`prisma/schema.prisma`）

"Schema"就是用 Prisma 的语法描述"有哪些表、每张表有哪些字段、表之间什么关系"。核心几张表：

```
User            用户（登录用户名、好友码、昵称、密码哈希、头像、在线状态字段…）
 ├─ ApiCredential   API 密钥（加密密文、末4位、日限额）
 ├─ LibraryEntry    库条目（用户获取了哪些产品、使用时长）
 ├─ FriendEdge      好友关系（aId↔bId，status: pending/accepted）
 ├─ FriendNote      好友备注（我给某好友起的别名）
 ├─ Message         私聊消息（fromId, toId, body, at）
 ├─ AchievementUnlock  成就解锁记录
 └─ UsageRecord     用量记录（哪个应用花了多少 token）

Product         产品（应用/模型/Agent/Skill/教程/视频，多态用 type 字段区分）
 ├─ Review          评测
 ├─ Achievement     成就定义（开发者配置的成就 schema）
 └─ status/ownerUserId/clientId  发布态、归属开发者、开放API凭证

OAuthGrant      OAuth 授权（外部应用的授权码与访问令牌）
```

### 设计要点①：多态产品（一张表装六种东西）
应用、模型、Agent、Skill、教程、视频都存在**同一张 `Product` 表**，用一个 `type` 字段区分。它们共享详情页、评分、评论、库、动态这套通用能力。

> **面试金句**：我把六类"造物"统一成一个多态 Product 模型——它们形态不同但都需要详情页/评分/获取/动态，统一建模避免了六套重复代码。这是 Steam 把游戏/DLC/软件/原声碟统一成"商品"的思路。

### 设计要点②：好友关系用"边"建模
`FriendEdge` 表每行是一条"边"（aId 和 bId 两个用户 + status）。`status` 为 `pending`（A 申请了 B，待 B 同意）或 `accepted`（已是好友）。查"我的好友"就是查"所有 accepted 的、包含我的边"。

```ts
const edges = await prisma.friendEdge.findMany({
  where: { status: "accepted", OR: [{ aId: me }, { bId: me }] },
});
```

## 3. "故意冗余"：去规范化（denormalization）

数据库教科书讲"规范化"（同一份数据只存一处，避免冗余）。但实际工程里**有时故意冗余能换性能/正确性**。我有两处典型例子：

### 例①：评测里冗余存了作者昵称
`Review` 表里除了 `authorUserId`（关联到用户），还**额外存了一份 `authorName` 和 `avatarHue`**。为什么不每次查评测时再去 join 用户表拿昵称？
- 部分评测作者是"种子数据/Agent"，根本没有真实用户行。
- **评测是"当时的快照"**——你写评测时叫什么名字，就该一直显示那个名字。
- 当你在资料页改昵称时，我会**同步更新你历史评测里的署名**（`profile-service.updateProfile` 里一个事务同时改 User 和 Review）。

> **面试金句**：评测作者名我做了**去规范化**——存一份快照而不是每次 join。代价是改名时要同步更新，但换来了"无主用户也能有评测"的灵活性和查询的简单。冗余不是错，是有意识的权衡。

### 例②：评分的"基数混合"
产品显示"2891 篇评测、4.9 分"，但数据库里其实只存了几条样例评测行。如果用户新写一条评测就重新统计行数，评测数会从 2891 **塌缩到个位数**，很穿帮。

我的解法：存一个 `baseRatingScore` / `baseRatingCount`（上架自带的"社会证明"基数），用户的真实新评测在这个基数上**加权混合**：

```
显示评测数 = base 基数 + 平台内真实评测行数
显示评分   = (base分 × base数 + 真实评测分数之和) / 显示评测数
```

这样你加一条 4 星评测，2891 变 2892，均分几乎不动——符合直觉。

> **面试金句**：种子的评分是"社会证明"展示值，不是真行数。我引入 base 基数，把新评测**混合**进去而非重新统计，避免了"用户一评测，评测数从几千塌成个位"的穿帮。这是处理"假数据 + 真数据共存"的实用技巧。

## 4. 数据库迁移（migration）

"迁移"= 当你改了 schema（加字段、加表），要把这个变更**安全地应用到已有数据库**，并留下变更历史（每次迁移一个带时间戳的 SQL 文件）。

正常用 `prisma migrate dev`，但我踩了个坑：

### 坑：`prisma migrate dev` 拒绝在非交互环境运行
它要弹个确认框等你按 y，但自动化脚本里没有交互终端，它就直接报错退出。

### 解法：用 `migrate diff` 生成 SQL，再 `migrate deploy` 应用

```bash
# ① 对比"当前数据库"和"目标 schema"，生成差异 SQL，写进迁移文件
prisma migrate diff --from-config-datasource prisma.config.ts \
  --to-schema prisma/schema.prisma --script > 迁移目录/migration.sql

# ② 非交互地应用所有未应用的迁移
prisma migrate deploy
```

`migrate deploy` 是为"生产部署"设计的，本来就非交互，不会弹确认。

> **面试金句**：`migrate dev` 是交互式的、适合本地开发；自动化/CI 里要用 `migrate diff` 生成 SQL + `migrate deploy` 应用——后者非交互、幂等、专为部署设计。理解这两个命令的分工是用好 Prisma 的关键。

## 5. 种子脚本与"防清库"（重要的运维教训）

"种子（seed）"= 往空数据库灌一批初始演示数据（用户、产品、评测…）。

### 我犯过的错 → 数据丢失
最初 `seed.ts` 是"**先 deleteMany 清空所有表，再重新插入**"。开发过程中我多次跑 `pnpm db:seed` 刷新演示数据，结果：
1. 把用户**注册的真实账号删了**。
2. 连种子用户的内部 `id` 都换了——导致用户登录用的 Cookie 指向一个不存在的 id，被当成未登录。

### 修复：默认非破坏性 + 显式重置
```ts
async function main() {
  const force = process.env.SEED_FORCE === "1";
  const existing = await prisma.user.count();
  if (existing > 0 && !force) {
    console.log("数据库已有数据，跳过种子（不清库）。如需清库重灌：pnpm db:reset");
    return;   // ← 有数据就直接跳过，绝不清库
  }
  // 只有空库或显式 SEED_FORCE=1 才会清库重灌
  await prisma.user.deleteMany(); ...
}
```

- `pnpm db:seed` → 有数据就跳过，**保护已注册账号**。
- `pnpm db:reset`（内部 `SEED_FORCE=1`）→ 才会真正清库重灌。

> **面试金句**：种子脚本默认必须是**幂等且非破坏性**的——有数据就跳过，清库要显式触发。我吃过"开发时随手 reseed 把测试账号清了、还顺带让登录态失效"的亏，之后把"破坏性操作"和"日常操作"严格分开。这是很实在的运维教训。

---

## 小结

- **Prisma + PostgreSQL**：用代码而非 SQL 操作库，连接用 globalThis 单例复用。
- **多态 Product**：一张表 + type 字段装六类造物，共享通用能力。
- **去规范化**：评测署名快照、评分基数混合——有意识地用冗余换正确性和性能。
- **迁移**：自动化用 `migrate diff` + `migrate deploy` 绕过交互限制。
- **种子**：默认非破坏性，清库要显式 `db:reset`——血泪教训。
