# 面试速记（一问一答）

> 面试前过一遍。每题先给"30 秒口述版"，再给"如果追问"的要点。配合前面 6 篇深入文档食用。

---

## 项目总览类

**Q：简单介绍下这个项目。**
> 星港是一个仿 Steam 的「AI 应用 + Agent 社交平台」。用户能逛商店、获取应用、配置 API 密钥后在站内应用里直接用、加好友实时聊天、解锁成就；开发者能上架应用、定义成就、通过开放 API 让外部 web 应用接入。技术上是 Next.js 16（App Router）+ React 19 + PostgreSQL（Prisma）的全栈单体，登录用 iron-session，实时聊天用短轮询，API 密钥用 AES-GCM 信封加密。

**Q：为什么选 Next.js 全栈单体，而不是前后端分离？**
> 一个人要快速把完整产品做出来，全栈单体省掉了"定义接口、跨服务通信"的开销。Next.js 的服务端组件能直接查库渲染首屏、Server Actions 省掉手写 REST，开发效率很高。如果团队变大、某模块要独立伸缩，再按模块拆服务——我从第一天就把业务逻辑收口在 `lib/` 的 service 层，拆分有清晰切线。

**Q：服务端组件和客户端组件的区别？**
> 服务端组件在服务器跑、能直接 await 查库、不给浏览器发 JS、但不能交互；客户端组件（"use client"）在浏览器跑、能用 useState/onClick/拖动、但要数据得通过 Server Action。典型套路是服务端组件查好初始数据传给客户端组件做后续交互。

## 实时聊天类（重点，必考）

**Q：聊天的实时是怎么实现的？**
> 短轮询。客户端每 2 秒带一个时间戳游标调一个 Server Action，服务器只返回"游标之后、发给我的新消息"加最新好友/在线状态。发消息用乐观更新让界面零延迟。新消息若对应会话没开着，就未读 +1 并弹提醒 Toast。

**Q：为什么不用 WebSocket？**（高频追问）
> 三点：① 需求只是"看到消息"，2 秒延迟够用，不需要毫秒同步；② 轮询无连接生命周期、无内存状态，数据库当中转，天然跨进程/多实例可靠，和无状态架构更契合；③ 我最初做过 SSE + 进程内事件总线，但 Next 生产构建会缓冲流式响应、且 Server Action 和路由不在同一 bundle 共享内存总线，导致推送丢失——这让我意识到"进程内状态"在这套架构里不可靠。如果将来要做"正在输入/已读/毫秒同步"，我会上 WebSocket + Redis Pub/Sub 把状态外置。

**Q：什么是乐观更新？**
> 先乐观假设操作会成功，立刻在本地更新 UI，再去后台真正写库，让交互零延迟；失败再回滚。我这里因为"收件箱查询天然排除自己发的消息"，连去重都不用。

**Q：轮询怎么不漏消息、不重复？**
> 用游标（时间戳书签）：每次先记下服务器当前时间 now，查 (上次游标, now] 区间发给我的消息，下次从 now 继续。自己发的消息 toId 是对方，不会被自己的收件箱查询拉回来，所以不重复。

**Q：定时器里读不到最新 state 怎么办？**
> 闭包陷阱——闭包里的 state 被定格成创建时的值。用 ref（`ref.current` 永远是最新值）来读"当前激活的会话"这类需要实时的值。

## 安全类（必考）

**Q：密码怎么存？**
> scrypt 加盐慢哈希。盐防彩虹表（相同密码不同哈希），scrypt 慢且耗内存防 GPU 暴力，比对用恒定时间比较防计时攻击。绝不存明文或可逆加密。

**Q：用户的 API Key 怎么保证安全？**（项目亮点）
> AES-256-GCM 信封加密：用环境变量里的主密钥加密 Key，密文存库——数据库泄露也解不开；GCM 的认证标签防篡改。解密只在服务端调用大模型那一瞬发生，第三方应用和前端只拿"调用结果"，全程接触不到明文 Key，界面也只显示末 4 位。

**Q：登录态怎么存？为什么安全？**
> iron-session 加密签名的 httpOnly Cookie。加密让内容保密、签名防篡改、httpOnly 防 XSS 偷 Cookie、sameSite 防 CSRF。服务端无状态，Cookie 本身就是凭证。

**Q：外部应用接入怎么做的？**
> OAuth2 授权码流：用户在平台授权 → 应用拿短命 code 在后端用 client_secret 换 access_token → 带 token 调 REST 接口，按 scope 鉴权。和"用 GitHub 登录"同一套，第三方拿有限权限令牌，碰不到用户密码。

**Q：平台内第三方应用怎么隔离？**
> sandbox iframe（不给 same-origin，读不到宿主 Cookie/DOM）+ postMessage RPC 受控通道。应用调 SDK 请求能力，宿主转发到网关、密钥服务端解密，应用只拿结果。和浏览器扩展、小程序的安全模型同源。

## 数据库类

**Q：什么是 ORM？为什么用 Prisma？**
> ORM 让你用写代码的方式操作数据库，不手写 SQL，有类型检查和自动补全。Prisma 还有 schema 驱动的迁移、生成类型安全的客户端。

**Q：讲一个你做的数据建模权衡。**
> 评分基数混合：种子里产品显示"2891 评测/4.9 分"，但只存了几条样例评测行。直接按行数统计会让用户一评测就塌缩到个位。我存一个 base 基数，把真实新评测加权混合进去，2891→2892、均分几乎不动。这是处理"假数据+真数据共存"的去规范化技巧。

**Q：迁移怎么做的，踩过坑吗？**
> `prisma migrate dev` 是交互式的，自动化环境会卡在确认框。改用 `migrate diff` 生成 SQL + `migrate deploy` 非交互应用。还吃过"种子脚本默认清库把测试账号删了、连登录态都失效"的亏，之后把种子改成默认非破坏性、清库要显式 `db:reset`。

## 工程/调试类

**Q：你 debug 过印象深的问题？**
> 发消息时报 "Cannot update a component while rendering"。根因是我在 setState 的更新函数里调了会触发路由更新的 Server Action——更新函数必须纯、不能有副作用。把异步加载挪出更新函数、用 ref 做去重守卫就好了。心法是"渲染要纯，副作用放事件处理器或 useEffect"。

**Q：实时这种功能你怎么测？**
> 用 Playwright 开两个独立浏览器上下文，各带不同用户的登录 Cookie，A 发消息断言 B 不刷新能收到、关闭会话时弹提醒。涉及两个客户端的功能必须端到端验证。一个坑：Playwright 的 goto 默认等 networkidle，但轮询让网络永不空闲，要改用 domcontentloaded。

**Q：项目里体现了哪些设计模式/原则？**
> 门面模式（lib/catalog 是数据唯一入口）、适配器模式（网关抹平各家大模型 API）、单一事实来源（"我是谁"收口到 getSessionUserId）、去规范化权衡、关注点分离（页面展示 / lib 业务逻辑 / Prisma 数据访问三层）。

## 加分回答（体现成长性思维）

**Q：这个项目还有什么可以改进/演进？**
> ① 实时从轮询升级到 WebSocket + Redis，做"正在输入/已读"；② 头像等从 dataURL 存库改成对象存储 + CDN；③ 网关接更多 provider、做更细的限流和成本告警；④ 桌面化（Tauri/Electron 壳 + 系统通知）；⑤ 按模块从单体拆出独立服务（网关是安全+高吞吐路径，最该先拆）。
