## 1. 可拖动 + 记住位置的聊天窗

### 需求
聊天窗初始居中；用户拖动后记住位置；关掉再打开、甚至刷新页面，都回到上次拖到的地方。

### 怎么做：指针事件 + localStorage

**第一步：用 React state 控制窗口位置。** 不再用固定的 CSS `right/bottom`，而是用 `style={{ left: pos.x, top: pos.y }}`，`pos` 是个 state。

**第二步：初始位置——记忆优先，否则居中。** 组件挂载时跑一次：

```ts
useEffect(() => {
  const saved = localStorage.getItem("starport_chat_pos");
  if (saved) { setPos(clamp(JSON.parse(saved))); return; }   // 有记忆 → 用它
  // 没记忆 → 居中
  setPos(clamp({ x: (window.innerWidth - W) / 2, y: (window.innerHeight - H) / 2 }));
}, []);
```

- **`localStorage`**：浏览器的本地小仓库，能跨刷新、跨开关持久存一点数据（key-value）。这就是"记住位置"的存储。
- **`clamp`（钳制）**：把位置限制在屏幕范围内，防止窗口被拖出可视区域消失。

**第三步：拖动——指针事件三件套。**

```ts
const onDragStart = (e) => {
  dragRef.current = { dx: e.clientX - pos.x, dy: e.clientY - pos.y };  // 记下鼠标相对窗口左上角的偏移
  e.currentTarget.setPointerCapture(e.pointerId);   // 关键：捕获指针，拖出元素也不丢事件
};
const onDragMove = (e) => {
  if (!dragRef.current) return;
  setPos(clamp({ x: e.clientX - dragRef.current.dx, y: e.clientY - dragRef.current.dy }));
};
const onDragEnd = () => {
  if (拖过) localStorage.setItem("starport_chat_pos", JSON.stringify(posRef.current));  // 落盘记忆
  dragRef.current = null;
};
```

三个细节：
- **为什么记 `dx/dy` 偏移**：不然窗口会"跳"到鼠标位置。记下"按下时鼠标在窗口里的相对位置"，移动时减掉它，窗口才跟手。
- **`setPointerCapture`**：把后续的 move/up 事件都"绑"到拖动条上，这样你拖得很快、鼠标离开了拖动条，事件也不丢。这是稳健拖拽的关键 API。
- **`touch-none`（CSS）**：禁掉触屏的默认手势，让手机上也能拖。

**第四步：为什么"关掉再开"能恢复？** 因为所有聊天关掉时，聊天窗组件会卸载（`openChats` 为空 → 返回 null）；重新打开时组件**重新挂载**，又跑一遍上面的 `useEffect`，从 localStorage 读回位置。

> **面试金句**：拖拽我用原生 **Pointer Events + `setPointerCapture`**（不依赖拖拽库），位置存 localStorage 实现跨会话记忆。`setPointerCapture` 保证快速拖动不丢事件，记 `dx/dy` 偏移保证跟手，clamp 防出界。组件卸载/重挂时从 localStorage 恢复，所以关了再开能回到原位。

## 2. 一个真实修过的 React bug（面试很爱聊"你 debug 过什么"）

### 现象
给好友发消息时控制台报错：
> `Cannot update a component (Router) while rendering a different component (SocialLayer)`

### 根因
打开会话时，我**在 `setState` 的更新函数里**调了一个会触发路由更新的 Server Action：

```ts
// ❌ 错误写法
setConversations((cur) => {
  if (cur[handle]) return cur;
  loadConversationAction(handle).then(...);  // ← 副作用！在更新函数里调 Server Action
  return { ...cur, [handle]: [] };
});
```

**React 的 state 更新函数必须是纯函数**——只能根据旧 state 算新 state，**不能有副作用**。React 在渲染期会调用它，而 `loadConversationAction` 触发了路由/请求更新，等于"在渲染一个组件的过程中去更新另一个组件"，React 就警告。

### 修复
把副作用**移出更新函数**，并用一个 `ref` 记录"哪些会话已经加载过"避免重复请求：

```ts
// ✅ 正确写法
const loadedRef = useRef(new Set());
const openChat = async (handle) => {
  setActiveChat(handle);
  if (!loadedRef.current.has(handle)) {        // 副作用在更新函数之外
    loadedRef.current.add(handle);
    setConversations(c => c[handle] ? c : { ...c, [handle]: [] });  // 纯：先占个空位
    const msgs = await loadConversationAction(handle);              // 副作用：单独 await
    setConversations(c => ({ ...c, [handle]: msgs }));
  }
};
```

> **面试金句**：那个报错的本质是"在渲染期产生副作用"。React 的 setState 更新函数必须纯。我把异步加载从更新函数里挪出来、用 ref 做去重守卫就解决了。这类 bug 的通用心法：**渲染要纯，副作用放进事件处理器或 useEffect**。

## 3. 头像上传：浏览器端压缩成 dataURL

### 问题
直接把用户选的原图（可能几 MB）传上去太大。而且我没有对象存储（OSS/S3）。

### 解法：浏览器端用 canvas 压缩 → 存成 dataURL

```ts
// src/components/settings/profile-form.tsx
function fileToAvatarDataUrl(file: File): Promise<string> {
  const img = new Image();
  img.src = URL.createObjectURL(file);
  await img.onload;
  const canvas = document.createElement("canvas");
  canvas.width = canvas.height = 256;               // 缩到 256×256
  const ctx = canvas.getContext("2d");
  // 居中裁剪成正方形再画进 canvas
  ctx.drawImage(img, sx, sy, min, min, 0, 0, 256, 256);
  return canvas.toDataURL("image/jpeg", 0.85);      // 导出成 JPEG dataURL（几十 KB）
}
```

- **`canvas`**：浏览器的画布，可以把图片画上去再导出——常用来做客户端图片处理（缩放、裁剪、加水印）。
- **dataURL**：把图片编码成一段 `data:image/jpeg;base64,....` 的文本字符串。好处是能直接当 `<img src>`，也能直接存数据库的文本字段，不需要文件存储服务。
- 压缩到 256×256 JPEG 后只有几十 KB，存进 `User.avatarUrl` 字段。服务端再校验它确实是 `data:image/...` 开头、且不超过大小上限。

> **面试金句**：头像上传我在**浏览器端用 canvas 居中裁剪 + 压缩成 256² JPEG dataURL**，再存数据库文本字段。好处是不依赖对象存储就能跑、上传体积可控；代价是 dataURL 比二进制大约 33%，所以只适合小图（头像），大图该上真正的对象存储 + CDN。

### Avatar 组件：图片优先，否则色相首字
`src/components/ui/avatar.tsx` 统一处理：有上传图就显示图，没有就显示"昵称首字 + 一个由色相生成的纯色底"。这样新用户不传头像也有个体面的默认头像。

## 4. 多 tab 会话 + 右键菜单（状态怎么组织）

聊天窗支持同时和多个好友聊、用 tab 切换。核心是几个 state（都在 `SocialLayer` 这个客户端组件里集中管理）：

```ts
const [openChats, setOpenChats] = useState<string[]>([]);       // 开着哪些会话（好友 handle 列表）
const [activeChat, setActiveChat] = useState<string | null>();  // 当前激活的 tab
const [conversations, setConversations] = useState<Record<...>>({});  // 每个会话的消息
const [unread, setUnread] = useState<Record<string, number>>({});     // 每个会话的未读数
```

- **双击好友** → 把它加进 `openChats` 并设为 `activeChat`，清掉它的未读。
- **tab 切换** → 改 `activeChat`。
- **右键好友** → 记下 `{ friend, x, y }`，在鼠标位置渲染一个固定定位的菜单（发消息/查看主页/设置备注/删除好友）；点页面任意处或滚动就关掉。

> **面试金句**：我把好友、会话、未读、右键菜单、提醒这些社交状态**集中在一个客户端编排组件（SocialLayer）**里管理，而不是散落在各处。轮询、事件、UI 都从这一个地方分发，状态流向清晰，避免了多组件各自持有状态导致的不一致。

## 小结

- **拖拽**：Pointer Events + setPointerCapture + localStorage 记忆，组件重挂时恢复。
- **React bug**：渲染期不能有副作用，setState 更新函数必须纯——把异步挪出去。
- **头像**：canvas 客户端压缩成 dataURL 存数据库，免对象存储。
- **状态组织**：社交状态集中在一个编排组件，状态流向清晰。
