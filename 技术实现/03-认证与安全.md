# 03 · 认证与安全

> 讲清楚：登录怎么"记住你"、密码怎么存才安全、用户的 API Key 怎么加密到"连跑在平台上的第三方应用都偷不走"。
> 涉及文件：`src/lib/session.ts`、`src/lib/password.ts`、`src/lib/crypto.ts`、`src/lib/auth-service.ts`。

## 1. 登录怎么"记住你"：iron-session

### 问题
HTTP 是**无状态**的——服务器处理完一个请求就忘了你是谁。那刷新页面后怎么还知道"你已登录"？

### 答案：加密的 Session Cookie
登录成功后，服务器给浏览器发一个 **Cookie**（浏览器会自动在之后每个请求里带上它）。Cookie 里存的不是明文"用户id=123"（那样能被篡改），而是一段**加密 + 签名**过的数据。这就是 `iron-session` 干的事。

```ts
// src/lib/session.ts
const sessionOptions = {
  password: process.env.IRON_SESSION_PASSWORD!,  // 一把 ≥32 字符的密钥，存环境变量
  cookieName: "starport_session",
  cookieOptions: {
    httpOnly: true,   // 关键：JS 读不到这个 Cookie，防 XSS 偷 Cookie
    sameSite: "lax",  // 防 CSRF（跨站请求伪造）
    secure: 由 COOKIE_SECURE 环境变量控制，默认生产为 true，  // 只走 HTTPS 传输
    maxAge: 30 天,
  },
};
```

> 关于 `secure`：纯 http 部署（如 `http://IP:3000` 自测）时必须设 `COOKIE_SECURE=false`，否则浏览器在 http 下不保存 Cookie，表现为「登录后又被要求登录」；上了 HTTPS 后再切回 `true` 更安全。详见 [07 部署与上线（HTTPS）](07-部署与上线（HTTPS）.md) 第 5 节。

- **加密**：Cookie 内容用 `IRON_SESSION_PASSWORD` 加密，浏览器和攻击者都看不懂里面是什么。
- **签名/防篡改**：你改一个字节，服务器解密就失败，会被当成无效。
- **`httpOnly`**：浏览器的 JavaScript **读不到**这个 Cookie——就算页面被注入恶意脚本（XSS），也偷不走你的登录态。

### 读"我是谁"

```ts
// 需要登录的页面/接口调它；没登录就跳转登录页
export async function getSessionUserId(): Promise<string> {
  const userId = await getSessionUserIdOrNull();
  if (!userId) redirect("/login");
  return userId;
}
```

整个项目"我是谁"全部收口到这一个函数。页面、Server Action、API 接口都调它——**单一事实来源**。

> **面试金句**：登录态我用 iron-session 存成**加密签名的 httpOnly Cookie**——加密让内容保密、签名防篡改、httpOnly 防 XSS 窃取、sameSite 防 CSRF。服务端不用维护 session 存储（无状态），Cookie 本身就是凭证。

## 2. 密码怎么存：scrypt 加盐哈希

### 铁律：密码**绝不能明文存**，也不能用可逆加密存。
万一数据库泄露，明文密码 = 所有用户裸奔。正确做法是存**哈希**（单向、不可逆）。

但光哈希还不够，要防两种攻击：
- **彩虹表**：攻击者预先算好"常见密码→哈希"的大表，一查就破。→ 用**加盐（salt）**：每个用户一个随机盐，哈希时把盐混进去，相同密码也得到不同哈希，彩虹表失效。
- **暴力破解**：GPU 一秒能算几十亿次普通哈希。→ 用**慢哈希函数**（scrypt/bcrypt/argon2）：故意算得慢且耗内存，让暴力破解代价高到不可行。

我用 Node 内置的 **scrypt**（`src/lib/password.ts`）：

```ts
export function hashPassword(password: string): string {
  const salt = randomBytes(16);                       // 每个用户一把随机盐
  const hash = scryptSync(password, salt, 64);        // 慢哈希
  return `${salt.toString("hex")}:${hash.toString("hex")}`;  // 存成 "盐:哈希"
}

export function verifyPassword(password: string, stored: string): boolean {
  const [saltHex, hashHex] = stored.split(":");
  const expected = Buffer.from(hashHex, "hex");
  const actual = scryptSync(password, Buffer.from(saltHex, "hex"), 64);  // 用同样的盐再算一遍
  return expected.length === actual.length && timingSafeEqual(expected, actual);  // 比对
}
```

两个细节：
- 盐和哈希一起存成 `盐:哈希`，验证时取出盐重新算、再比对。
- 用 `timingSafeEqual` 做**恒定时间比较**，防"计时攻击"（通过比较耗时差异推断密码）。

> **面试金句**：密码我用 **scrypt 加盐**存。盐防彩虹表（相同密码不同哈希），scrypt 的"慢且耗内存"防 GPU 暴力破解，比对用恒定时间比较防计时攻击。整个过程不引第三方依赖，用 Node 内置 crypto。

## 3. 重头戏：API Key 的"信封加密"

这是平台最核心的安全承诺：**你在平台配置一次 API Key（如 Claude），平台上任何应用都能经你授权用它，但谁都拿不到你的明文 Key——包括跑在平台上的第三方应用。**

### 怎么做到的？AES-256-GCM 信封加密（`src/lib/crypto.ts`）

```ts
const ALGO = "aes-256-gcm";
function getKey() {
  // 主密钥放环境变量，不进数据库、不进代码库
  return Buffer.from(process.env.STARPORT_SECRET, "hex");  // 32 字节
}

export function encryptSecret(plaintext: string): string {
  const iv = randomBytes(12);                          // 每次加密用随机 IV（初始向量）
  const cipher = createCipheriv(ALGO, getKey(), iv);
  const ciphertext = Buffer.concat([cipher.update(plaintext, "utf8"), cipher.final()]);
  const tag = cipher.getAuthTag();                     // GCM 的认证标签，防篡改
  // 存成 "iv.tag.密文" 三段
  return [iv.toString("base64"), tag.toString("base64"), ciphertext.toString("base64")].join(".");
}

export function decryptSecret(packed: string): string {
  const [iv, tag, data] = packed.split(".");
  const decipher = createDecipheriv(ALGO, getKey(), Buffer.from(iv, "base64"));
  decipher.setAuthTag(Buffer.from(tag, "base64"));     // 验证没被篡改
  return Buffer.concat([decipher.update(...), decipher.final()]).toString("utf8");
}
```

几个名词：
- **信封加密**：用一把"主密钥"（`STARPORT_SECRET`，存环境变量）加密你的 API Key，**密文存数据库**。数据库就算整个泄露，没有主密钥也解不开。
- **AES-256-GCM**：一种对称加密算法。GCM 模式额外带一个**认证标签（auth tag）**，既加密又防篡改——密文被改一个字节，解密就失败。
- **IV（初始向量）**：每次加密用一个随机 IV，保证"同样的 Key 加密两次得到的密文也不同"，更安全。

### 关键安全属性：明文只在服务端"一闪而过"

数据流（`src/lib/gateway-core.ts`）：

```
用户配置 Key ──► encryptSecret ──► 密文存数据库
                                       │
应用要调大模型时（在服务端）：           │
  从数据库取密文 ──► decryptSecret ──► 明文 Key（只在内存里存在这一瞬）
                                       │
                          用明文 Key 调 Anthropic/OpenAI
                                       │
                          调用结果返回给应用（应用全程没碰到明文 Key）
```

- 解密**只发生在服务器内存里、调用大模型的那一瞬间**。
- 前端、第三方应用，拿到的永远是"调用结果"，不是 Key。
- 列表展示只给你看**末 4 位**（`····7f3a`），连你自己在界面上也看不到完整 Key。

> **面试金句**：API Key 用 **AES-256-GCM 信封加密**存数据库——主密钥在环境变量里，数据库泄露也解不开；GCM 的认证标签防篡改。解密只在服务端调用大模型那一瞬发生，第三方应用和前端全程只能拿到"调用结果"，接触不到明文 Key。这是"配置一次全平台用"能成立的安全基础。

## 4. 路由守卫：哪些页面要登录

不同页面有不同要求，我在每个页面/接口入口做判断：

| 页面 | 规则 |
|---|---|
| 商店、产品详情 | **允许匿名**浏览（用 `getSessionUserIdOrNull()`，没登录返回 null，照常展示） |
| 库、个人主页、设置、启动应用 | **必须登录**（用 `getSessionUserId()`，没登录直接 `redirect("/login")`） |
| 开放 API `/api/v1/*` | 必须带合法的 access token，否则返回 401 |

导航栏也据此切换：登录了显示头像 + 设置；没登录显示"登录"按钮。

> **面试金句**：我区分"匿名可浏览"和"必须登录"两类入口，用 `getSessionUserId()`（强制登录、自动跳转）和 `getSessionUserIdOrNull()`（容忍匿名）两个函数表达，守卫逻辑收口在 session 层，页面只管调。

## 5. 身份模型：用户名 / 昵称 / 好友码的区分

一个常见设计错误是"用展示名当唯一标识"。我把身份拆成三个字段（`src/lib/auth-service.ts`、`profile-service.ts`）：

| 字段 | 作用 | 能改吗 | 唯一吗 |
|---|---|---|---|
| `handle` 用户名 | 登录用的标识 | ❌ 不可改 | ✅ 唯一 |
| `name` 昵称 | 界面展示名 | ✅ 随时改 | ❌ 可重名 |
| `friendCode` 好友码 | 加好友用的辨识码（`SP-7K3M9Q`） | ❌ 不可改 | ✅ 唯一 |

为什么这么拆？
- **昵称可重名**，所以不能拿它做识别——这正是你提的"用户名会重名"的问题。
- **好友码唯一且不可改**，专门用来精确加人，且换昵称不影响。
- 好友码用一套**去歧义字符集**生成（去掉了 `0/O/1/I` 这种容易看错的），加好友时还做大小写/缺前缀的容错。

> **面试金句**：我把"登录标识、展示名、社交辨识码"三者分离——展示名可改可重名、好友码唯一不可改。这是 Steam、Discord 等成熟社交产品的通用做法，避免了"改个名字就找不到人"或"重名加错人"。

---

## 小结

- **登录**：iron-session 加密签名的 httpOnly Cookie，无状态，防 XSS/CSRF。
- **密码**：scrypt 加盐慢哈希 + 恒定时间比较，防彩虹表/暴力/计时攻击。
- **API Key**：AES-256-GCM 信封加密，主密钥在环境变量，明文只在服务端调用瞬间存在，应用永远拿不到。
- **身份**：用户名/昵称/好友码三分离，昵称可改可重名，好友码唯一不可改。
