# 02 · 实时聊天系统（重点）

> 这是你最关心的一篇。讲清楚：消息怎么做到**不刷新页面就能收到**、提醒怎么弹出来、为什么我没用 WebSocket、以及这套方案的取舍。
> 涉及文件：`src/components/social/social-layer.tsx`（前端编排）、`src/lib/message-service.ts`（消息逻辑）、`src/app/friends-actions.ts`（Server Actions）。

---

## 1. 先搞懂问题：什么叫"实时"？难在哪？

普通网页是"**你问、服务器答**"：你点一下、刷新一下，浏览器才去服务器要数据。

但聊天不一样：**别人给你发消息时，是"服务器主动有了新数据"，可你的浏览器并没在问。** 浏览器怎么知道有新消息？这就是实时通信要解决的核心矛盾——

> HTTP 协议天生是"客户端发起、服务器响应"的单向模式，服务器**没法主动推**给一个安静待着的浏览器。

业界有三种主流解法：

| 方案 | 原理 | 比喻 |
|---|---|---|
| **轮询 Polling** | 客户端每隔几秒主动问一次"有新消息吗" | 你每隔 2 秒往收发室跑一趟看有没有信 |
| **SSE (Server-Sent Events)** | 客户端开一条长连接，服务器有数据就顺着这条连接推下来（单向：服务器→客户端） | 你在收发室拉了根管道，有信他直接塞管道里给你 |
| **WebSocket** | 客户端和服务器建一条**双向**长连接，谁都能随时推 | 你俩之间架了条专线电话，随时通话 |

**星港最终用的是轮询。** 下面讲为什么——这部分面试很爱问"为什么不用 WebSocket"。

## 2. 我的技术选型过程（这段最值钱，体现工程判断）

### 第一次尝试：SSE + 进程内事件总线（失败了）

我一开始想做"真推送"，方案是 SSE：

1. 开一个流式接口 `/api/stream`，浏览器用 `EventSource` 连上它，保持长连接。
2. 进程里放一个**事件总线**（Node 的 `EventEmitter`，就是个"发布-订阅"中心）。
3. 当 A 给 B 发消息时，发消息的 Server Action 往总线 `publish(B, 消息)`；B 的 `/api/stream` 长连接 `subscribe(B)`，收到就顺着流推给 B 的浏览器。

```
A 发消息 → Server Action → 总线.publish(B 的id, 消息)
                                  │
                                  ▼
                        B 的 /api/stream 订阅着 B 的id
                                  │
                                  ▼
                        顺着 SSE 长连接推给 B 的浏览器
```

**为什么失败？** 我用 Playwright 开两个浏览器实测，B 发的消息 A 收不到。排查发现两个致命问题：

1. **`next start`（生产构建）会缓冲流式响应**：我往流里写的初始事件，curl 测 3 秒一个字节都没收到——框架/运行时把流缓冲住了，没有立即 flush 给客户端。
2. **进程内总线在生产构建里不共享**：Next 会把"Server Action 的代码"和"路由接口的代码"打包成**不同的代码块（bundle）**。`realtime.ts` 这个模块被两边分别 import，可能各自实例化了一个 `EventEmitter`——于是 `publish` 往 A 总线发、`subscribe` 在 B 总线听，**两个总线根本不是同一个**，消息自然丢了。

> **面试金句**：SSE + 进程内 EventEmitter 在"单进程长驻"的假设下才成立。但 Next.js 生产构建会拆 bundle、会缓冲流式响应，而且真实部署常是 serverless / 多实例——**进程内的内存状态不可靠**。这是我踩的坑，也是我换方案的理由。

### 第二次：短轮询（成功，最终方案）

既然"服务器主动推"在这套架构里不可靠，那就反过来：**让客户端每 2 秒主动来拉**。

为什么轮询在这里是更好的工程选择：

- **无连接生命周期**：不用管长连接断了怎么重连、缓冲、心跳。
- **无内存状态**：不依赖"进程一直活着、总线是同一个"。每次轮询就是一次普通的 Server Action 调用，**数据源是数据库**，天然跨进程、跨实例可靠。
- **够用**：2 秒延迟，人聊天体感上就是"立刻看到"。

代价是：有 2 秒延迟、有一点点空轮询开销。对这个产品完全可接受。

> **面试金句**：我从 SSE 切到轮询，不是因为轮询更高级，而是因为它**和这套无状态、可水平扩展的架构更契合**。技术选型要服务于架构约束，而不是追新。如果将来要做"打字中…"或毫秒级同步，我会上 WebSocket，并把状态外置到 Redis 这类共享存储。

## 3. 轮询方案是怎么实现的（拆开讲）

核心就三件事：**一个游标、一个定时拉取、增量返回**。

### 3.1 关键概念：游标（cursor）

游标就是一个**时间戳书签**，记录"我已经看到截止到什么时候的消息了"。每次轮询带上这个书签，服务器只返回"比书签更新的"消息——这叫**增量拉取**，避免每次把全部历史都搬一遍。

前端用一个 `ref` 存这个游标（`src/components/social/social-layer.tsx`）：

```ts
const sinceRef = useRef<string>(new Date().toISOString());  // 初始 = 页面打开的时刻
```

### 3.2 客户端：每 2 秒拉一次

```ts
useEffect(() => {
  let alive = true;
  let timer;
  const tick = async () => {
    try {
      // 带着游标去拉：只要比 sinceRef 更新的东西
      const res = await pollUpdatesAction(sinceRef.current);
      if (!alive) return;
      sinceRef.current = res.now;        // 游标前移到服务器这次的时间
      setFriends(res.friends);           // 顺便刷新好友/在线状态
      setRequests(res.requests);         // 顺便刷新好友请求
      for (const m of res.messages) {    // 逐条处理新消息
        appendToConversation(m);         // 加到对应会话
        if (该会话没开着) { 未读+1; 弹提醒toast; }
      }
    } catch { /* 网络抖动，忽略，下次再来 */ }
    finally { if (alive) timer = setTimeout(tick, 2000); }  // 2 秒后再来一次
  };
  timer = setTimeout(tick, 2000);
  return () => { alive = false; clearTimeout(timer); };  // 组件卸载时停掉
}, []);
```

注意几个工程细节：
- **用 `setTimeout` 链式调用，不用 `setInterval`**：保证"上一次拉完才排下一次"，避免网络慢时请求堆叠。
- **`alive` 标志位**：组件卸载后不再 setState，避免 React 警告和内存泄漏。
- **`try/catch` 吞掉网络抖动**：一次失败不影响下一轮。

### 3.3 服务端：增量返回（一个 Server Action）

```ts
// src/app/friends-actions.ts
export async function pollUpdatesAction(since: string): Promise<PollResult> {
  const now = new Date().toISOString();          // 先记下"现在"，作为下次的游标
  const [messages, friends, requests] = await Promise.all([
    getIncomingSince(since),       // 只查 since 之后、发给我的消息
    getFriendsWithPresence(),      // 好友列表（含在线状态）
    getIncomingRequests(),         // 待处理好友请求
  ]);
  return { now, messages, friends, requests };
}
```

`getIncomingSince` 的核心就是一句"查比游标新的、收件人是我的消息"（`src/lib/message-service.ts`）：

```ts
export async function getIncomingSince(sinceIso: string) {
  const userId = await getSessionUserId();        // 我是谁（从 Cookie 解出来）
  const rows = await prisma.message.findMany({
    where: { toId: userId, at: { gt: sinceIso } }, // 收件人=我，且时间 > 游标
    orderBy: { at: "asc" },
    select: { body: true, at: true, from: { select: { handle: true, name: true } } },
  });
  return rows.map(r => ({ from: r.from.handle, fromName: r.from.name, body: r.body, at: r.at }));
}
```

> **为什么游标取"服务器的 now"而不是"最后一条消息的时间"？** 为了不漏消息：先记下 `now`，再查 `(since, 现在]` 区间的消息，下次从 `now` 继续。简单且不会因为某条消息时间戳相同而重复或遗漏。

### 3.4 数据流全景图

```
       A 的浏览器                          服务器 / 数据库                      B 的浏览器
          │                                                                       │
          │  发消息（乐观更新：先本地显示）                                          │
          │  sendMessageAction("B","你好") ───► 写入 Message 表（toId=B）           │
          │                                                                       │
          │                                          每 2 秒 ◄── pollUpdatesAction(since) ─┤
          │                                          查 toId=B 且 at>since 的消息           │
          │                                          找到「你好」───────────────────►│
          │                                                                       │ 显示「你好」
          │                                                                       │ （该会话没开 → 未读+1 + 弹提醒）
```

关键点：**A 和 B 之间没有任何直接连接，全靠数据库当"中转站"**。A 写库，B 轮询读库。这就是为什么这套方案天然可靠、可扩展——数据库本来就是多进程共享的。

## 4. 发消息：乐观更新（Optimistic UI）

如果你发一条消息，要等服务器写完库、等下次轮询才显示，会有明显延迟、手感很差。所以用**乐观更新**：

> 乐观更新 = "乐观地假设操作会成功"，**先在本地立刻把消息显示出来**，再去后台真正写库。

```ts
// src/components/social/social-layer.tsx
const sendChat = async (handle, body) => {
  const optimistic = { from: "me", body, at: new Date().toISOString() };
  // ① 立刻把自己的消息塞进会话——界面瞬间就有了
  setConversations(c => ({ ...c, [handle]: [...(c[handle] ?? []), optimistic] }));
  // ② 再真正写库（用户感知不到这个延迟）
  await sendMessageAction(handle, body);
};
```

**为什么轮询不会让我自己的消息重复显示？** 因为 `getIncomingSince` 只查 `toId = 我` 的消息（别人发给我的）。我自己发出去的消息 `toId` 是对方，不会被我自己的轮询拉回来。所以"自己发的"只靠乐观更新显示一次，不重复。

> **面试金句**：乐观更新是聊天/点赞这类高频交互的标配——先改 UI 再发请求，让界面零延迟；失败了再回滚。我这里因为"收件箱查询天然排除了自己发的消息"，所以连去重都不用做。

## 5. 消息提醒（未读 + Toast）是怎么来的

轮询拿到新消息后，判断"这条消息对应的会话是不是正开着且是当前激活的 tab"：

- **正在看这个会话** → 直接把消息追加进去，不打扰。
- **没开 / 不是当前 tab** → 未读数 +1（坞按钮上显示小红点数字），并弹一个 **Toast 提醒卡片**（4.5 秒后自动消失，点一下直接打开该会话）。

```ts
if (activeChatRef.current !== conv) {          // 不是当前正看的会话
  setUnread(u => ({ ...u, [conv]: (u[conv] ?? 0) + 1 }));   // 未读+1
  const id = Date.now() + Math.floor(performance.now());    // 唯一 id
  setToasts(t => [...t, { id, fromName, body, handle: conv }]);  // 弹提醒
  setTimeout(() => setToasts(t => t.filter(x => x.id !== id)), 4500); // 自动消失
}
```

这里用 `activeChatRef`（一个 ref）而不是直接读 state，是因为这段代码在轮询的闭包里跑，需要读到"最新"的当前会话，而不是闭包创建时那一刻的旧值。

> **小知识：为什么用 ref 不用 state？** state 在闭包里会被"定格"成创建时的值（闭包陷阱）。ref 是一个稳定的引用，`ref.current` 永远读到最新值。定时器/订阅里要读最新 state，常用这个技巧。

## 6. 好友/在线状态/好友请求：顺便一起实时了

注意 `pollUpdatesAction` 不只返回消息，还返回 `friends` 和 `requests`。所以**同一个轮询顺手解决了好几件实时**：

- **加好友免刷新**：B 同意了 A 的请求，A 这边下一次轮询 `getFriendsWithPresence()` 就会查到 B 成了好友，列表自动更新。
- **在线状态实时**：好友上线/正在用某个应用，轮询拿到最新 presence，状态点颜色自动变。
- **好友请求红点**：别人加你，轮询拿到 `requests`，添加按钮上的红点自动出现。

> **面试金句**：我把"消息 + 好友 + 在线状态 + 请求"合并进一个轮询接口，一次往返全搞定，省请求数。这是轮询方案的额外好处——批量同步，而不是每种数据各开一条连接。

**在线状态是怎么判断的？**（`src/lib/friends-service.ts` 的 `derivePresence`）：每个用户存了 `lastSeenAt`（最近活跃时间）和 `currentActivity`（正在用什么应用）。登录后每次访问页面会 `touchPresence()` 更新 `lastSeenAt`。判断规则：
- 10 分钟内有 `currentActivity` → "正在使用 X"（绿色）
- 5 分钟内活跃过 → "在线"（蓝色）
- 否则 → 回退到种子设定的展示状态 / "离线"

## 7. 这套方案的取舍与"将来怎么演进"（面试加分）

**现在的取舍：**
- ✅ 简单、可靠、无连接管理、天然可水平扩展（数据库当中转）。
- ✅ 一个接口批量同步多种数据。
- ⚠️ 有最多 2 秒延迟。
- ⚠️ 有空轮询开销（没新消息也每 2 秒查一次）。

**如果要更实时 / 更省，怎么演进：**
1. **WebSocket + Redis Pub/Sub**：要做"对方正在输入…"、已读回执、毫秒级同步时上 WebSocket；把"谁在线、谁订阅了哪个房间"的状态放 **Redis**（外部共享存储），解决"多实例不共享内存"的问题。
2. **长轮询（long polling）**：服务器收到轮询请求后**hold 住**，有新消息才立刻返回，没有就等到超时——比定时短轮询更省、更快，是 WebSocket 之前的过渡方案。
3. **降低空轮询**：页面在后台标签页时拉长间隔（用 `document.visibilityState`），省电省流量。

> **面试金句**：我会按需求复杂度选方案——"看到消息"用轮询够了；"打字中/已读/毫秒同步"才值得上 WebSocket，并把会话状态外置到 Redis。**不为了用而用**。

## 8. 我是怎么验证它真的工作的（很重要）

实时功能光自己点一下证明不了——得有"两个人"。我用 **Playwright 开两个独立浏览器上下文**（各自带不同用户的登录 Cookie）：

1. 上下文 A 登录为"我"，打开和"临渊"的聊天窗。
2. 上下文 B 登录为"临渊"，给"我"发一条消息。
3. **不刷新** A，断言 A 的聊天窗里出现了那条消息 → `RESULT live: true`。
4. 关掉 A 的会话，B 再发一条 → 断言 A 弹出了提醒 Toast → `RESULT toast: true`。

> **面试金句**：实时这种"涉及两个客户端"的功能，我用多浏览器上下文做端到端验证，而不是只测单边逻辑——这才能真正证明"A 发、B 不刷新能收到"。

一个踩到的小坑：Playwright 的 `goto` 默认等 `networkidle`（网络空闲），但**轮询会让网络永远不空闲**，导致一直等不到。解决：改用 `domcontentloaded`（DOM 加载完即可），别等网络静默。

## 小结（一段话讲完）

> 星港的实时聊天用**短轮询**：客户端每 2 秒带一个**时间戳游标**调一个 Server Action，服务器只返回"游标之后、发给我的新消息"加上最新好友/在线状态。发消息用**乐观更新**让界面零延迟。我最初做的 SSE 因为 Next 生产构建缓冲流、且进程内事件总线不跨 bundle 共享而失败，于是切到轮询——它**无连接、无内存状态、靠数据库当中转**，和这套无状态可扩展的架构最契合。验证用 Playwright 双浏览器上下文跑通"A 发 B 免刷新收到"。
